# javascript review rules

## 必須チェック（優先）

- 未処理エラー（`try/catch` / `catch` 不足）
- 重い処理の直結（`scroll` / `resize` / `mousemove` など）
- 依存過多（標準APIで代替可能なライブラリ導入）
- 役割分離不足（モジュール責務の混在）

### 検出シグナル

- `await` 連鎖に失敗時ハンドリングがない
- イベントハンドラ内で重いDOM操作/計算を毎回実行
- 単一ファイルで取得・整形・描画・副作用を同時に実施
- 使っていない依存、または小機能のための大型依存
- タイマー/リスナーの解放漏れ

## 推奨チェック

- 読み込み戦略（`type="module"` / `defer`）
- 命名の明確性（意図が読める名前）
- 高頻度処理の最適化（Observer API, throttle/debounce）
- 本番配布の安定性（依存バージョン固定）

### 非同期処理の観点

- API失敗時のUIフォールバック（再試行・メッセージ表示）があるか
- `Promise.all` 利用時に一部失敗の扱いを定義しているか
- 中断可能な処理で `AbortController` を検討しているか

### パフォーマンスの観点

- レイアウトスラッシング（読み取り→書き込みの頻繁往復）を避けているか
- 高頻度イベントを `throttle` / `debounce` / `requestAnimationFrame` で制御しているか
- 不要な再レンダー/再計算を引き起こす状態更新がないか

## レビュー除外

- インデント、改行位置、クォート種別など整形論点
- Formatter/Linterで自動解決できるスタイル差分

## 出力テンプレ

- 位置: `file:line`（分かる範囲）
- 重要度: 重大 / 中 / 軽微
- 確度: 確定 / 推定 / 判定保留
- 現状:
- 問題:
- 修正案:

## 最小修正ガイド

- まず例外経路を埋める（`try/catch`、失敗時return）
- 次に高頻度イベント経路の負荷を制御
- 最後に責務分離（取得/変換/描画/副作用）を段階的に行う

## 判定保留の例

- 実行環境（ブラウザ対象・ビルド設定）が不明で最適戦略を断定できない
- 運用要件（許容遅延、再試行回数）が未定義

## 重大度ヒント

- 重大: 未処理例外・実行停止・高頻度イベント由来の深刻な性能劣化
- 中: 責務混在や読み込み戦略不備による不安定化
- 軽微: 命名・構造改善などの可読性向上提案
