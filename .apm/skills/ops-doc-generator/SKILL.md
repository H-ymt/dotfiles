# 運用ドキュメント生成スキル

Web サイト／Web アプリの **運用・保守ドキュメント** を生成するスキルです。
技術者ではないクライアント（病院、企業、自治体など）が読む前提で、平易な日本語で記述します。

## ワークフロー

### Step 0: リポジトリ自動調査

ドキュメント生成の前に、以下の情報源を **エージェントが自動で** 読み取り、使用サービス・構成を推定します。

| 調査対象 | 読み取るファイル例 |
|----------|-------------------|
| ホスティング | `wrangler.toml`, `vercel.json`, `vercel.ts`, `netlify.toml`, `docker-compose.yml` |
| CMS | `package.json` の依存（`microcms-js-sdk`, `contentful` 等）、`.env.example` の API キー名 |
| メール送信 | `.env.example`（`RESEND_API_KEY`, `SENDGRID_API_KEY` 等） |
| CI/CD | `.github/workflows/`, `cloudbuild.yaml` |
| セキュリティ | `.env.example`（`TURNSTILE_SECRET_KEY`, `RECAPTCHA_SECRET_KEY` 等） |
| 依存更新 | `.github/dependabot.yml`, `renovate.json` |
| 既存ドキュメント | `docs/`, `README.md`, `CLAUDE.md` |

調査が終わったら、**収集結果をまとめてユーザーに提示** します。

### Step 1: インタラクティブヒアリング

リポジトリからは取得できない情報をユーザーに質問します。
**一度にすべてを聞かず、カテゴリごとに段階的に進めます。**

#### ラウンド 1: 基本情報

- クライアント名（組織名）
- プロジェクト名 / サイト名
- 運用会社名
- ドキュメントの主な読者（クライアント担当者 / 社内チーム / 両方）

#### ラウンド 2: サイト・URL

- 本番 URL（サイトごと）
- ステージング URL（サイトごと）
- ステージングのアクセス制限方式と追加手順

#### ラウンド 3: サービス契約情報

Step 0 で検出したサービスごとに:
- 契約アカウント（メールアドレス）
- プラン・月額料金
- 管理画面 URL（自動推定できない場合）

#### ラウンド 4: メール・通知

- フォームの送信元アドレス（サイトごと）
- 通知先アドレス（ステージング / 本番、フォーム種別ごと）
- 監視アラートの設定有無・通知先

#### ラウンド 5: 権限・障害対応

- サービスごとのアカウント運用方針（個別招待 / 共通アカウント）
- エスカレーションフロー（一次対応の担当）
- ドメインの管理者（自社 / 別会社）

**ルール:**
- 各ラウンドの回答を受け取ってから次に進む
- 自動収集できた項目は確認として提示し、「合っていますか？」と聞く
- ユーザーが「わからない」「後で確認する」と答えた項目は `{{要確認: ○○}}` プレースホルダーで残す
- ユーザーが「このセクションは不要」と言ったら、以降のラウンドでその関連質問をスキップする

[ヒアリング項目の全量リスト](references/hearing-checklist.md) も参照。

### Step 2: ドキュメント生成

すべてのヒアリングが完了したら、[ドキュメントテンプレート](references/template.md) に沿って Markdown を生成します。

- 出力先はユーザーに確認する（例: `docs/運用ドキュメント.md`）
- 該当しないセクションは省略
- サービス固有の注意事項は [サービスカタログ](references/service-catalog.md) を参考に記載

### Step 3: レビュー・調整

生成したドキュメントをユーザーに確認してもらい、修正を加えます。
「全体を見て、直したいところがあれば教えてください」と伝える。

## ドキュメント構成（標準）

| # | セクション | 内容 |
|---|-----------|------|
| 1 | 本ドキュメントについて | 目的・対象読者・関連リポジトリへのリンク |
| 2 | サイト一覧 | 本番 URL・ステージング URL・アクセス制限の説明 |
| 3 | 利用サービス一覧 | 各サービスの用途・プラン・管理画面 URL・注意事項 |
| 4 | 権限・アカウント管理 | アカウント運用方針・メンバー追加/削除手順 |
| 5 | コンテンツ更新手順 | CMS の操作手順・プレビュー・自動デプロイの説明 |
| 6 | デプロイの仕組み | CMS 更新時・コード変更時のデプロイフロー |
| 7 | 依存関係の更新 | Dependabot / Renovate の対応手順（開発者向け） |
| 8 | 障害時の確認方法 | 症状別チェックリスト・ステータスページ・エスカレーション |
| 9 | 変更履歴 | ドキュメントの更新記録 |

**プロジェクトに該当しないセクションは省略**します（例: CMS を使っていなければ「5. コンテンツ更新手順」は不要）。

## 記述ガイドライン

- **読者は非エンジニア**を想定。専門用語には補足を添える
- 表形式を積極的に使い、視認性を高める
- 各セクションは独立して読めるようにする
- 「まず確認すること」→「それでもダメなら運用会社に連絡」のフローを明示
- 環境ごと（ステージング / 本番）の違いを明確に区別
- ランニングコストは「目安」と明記し、正確な金額は各ダッシュボードを参照するよう案内
- ユーザーが回答できなかった項目は `{{要確認: ○○}}` で残し、後から埋められるようにする
