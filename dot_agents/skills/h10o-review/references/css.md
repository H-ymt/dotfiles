# css review rules

## 必須チェック（優先）

- 責務分離（見た目/状態/レイアウトの混在を抑える）
- 詳細度の暴走回避（過度なネスト、要素依存の回避）
- 重複定義の整理（同じ意図の値・宣言の重複）
- トークン化可能な値の統一（色・間隔・サイズなど）

### 検出シグナル

- 同一色・同一余白が複数箇所でハードコード
- `!important` の常用
- `#id .class tag` のような強いセレクタ依存
- コンポーネント境界を跨ぐ指定（親依存が強すぎる）
- 似たUIを別クラスで都度再実装
- `z-index` の場当たり値（例: `999`）や用途不明な直指定

## 推奨チェック

- BEMなど一貫した命名
- `transition` 対象の明示（`all` 常用回避）
- `z-index` は原則 `1/-1` を直接指定し、例外値はカスタムプロパティ経由で管理しているか
- Tailwind利用時の責務分離（ユーティリティと生CSSの重複回避）

### カスケードレイヤーの観点

- `@layer` 宣言でレイヤー順序がファイル冒頭に明示されているか
- サードパーティ CSS（リセット・ライブラリ）が専用レイヤーに隔離されているか
- レイヤー内での `!important` 使用を避けているか（レイヤー優先順位が逆転するため）
- コンポーネント CSS が適切なレイヤーに属し、グローバルスタイルと分離されているか

### Tailwind併用時の観点

- 同一要素で「ユーティリティ」と「同等の生CSS」を二重定義していないか
- `@apply` は原則不使用としているか
  - 許容: マークアップにクラスを付与できない箇所（擬似要素、サードパーティ要素など）
  - 非推奨: ユーティリティの集約や可読性改善目的（Tailwind のユーティリティファーストの利点を損なう）
- デザイントークン（色・spacing・radius）が設定に寄せられているか
- ブレークポイント方針（`sm/md/lg`）が場当たりで増えていないか

### モーション/描画性能の観点

- `transition: all` を避け、`opacity` / `transform` へ限定しているか
- スクロール連動の重い視覚効果（過度な `filter` / `backdrop-filter`）がないか
- 不要な再計算を招く複雑セレクタを多用していないか

## レビュー除外

- インデント、空白、改行位置
- 宣言順の細かな好み（自動整形で吸収できる範囲）

## 出力テンプレ

- 位置: `file:line`（分かる範囲）
- 重要度: 重大 / 中 / 軽微
- 確度: 確定 / 推定 / 判定保留
- 現状:
- 問題:
- 修正案:

## 最小修正ガイド

- 値の重複はトークン（CSS変数/Tailwind設定）へ集約
- セレクタ詳細度は1段階ずつ弱める
- `z-index` は原則 `1/-1`、例外値はカスタムプロパティで管理
- 詳細度の衝突は `@layer` 導入でレイヤー順序に委ねる（セレクタ強化より優先）
- サードパーティ CSS は `@layer vendor` 等に隔離し、プロジェクト CSS との干渉を防ぐ
- 既存のUI崩れ回避のため、まず局所修正→必要時に再設計

## 判定保留の例

- デザイン仕様未共有で色や余白の妥当性を断定できない
- 画面遷移/JS制御を見ないと状態クラスの是非を断定できない

## 重大度ヒント

- 重大: 表示崩れ直結、保守不能化を招く詳細度/依存の暴走
- 中: 重複・責務混在により改修コスト増大
- 軽微: トークン統一や設計整流化などの改善提案
