## 7. 依存関係の更新（Dependabot）

> この節は **運用会社（開発担当）向け** です。

各リポジトリでは {{Dependabot / Renovate}} が {{更新頻度}}、依存パッケージの更新 PR を自動作成します。
放置すると PR が溜まって対応が大変になるため、{{推奨頻度}}は開いている PR を確認・マージすることを推奨します。

### 7.1 対応の流れ

1. GitHub から通知が届く（例: 「Bump next from 15.2.3 to 15.2.4」）
2. リリースノートを確認（破壊的変更や非推奨化がないか）
3. CI の結果を確認（緑のチェックマーク = 自動ビルド・チェックが通っている）
4. develop にマージ → ステージング確認 → 本番反映

| CI 結果 | 更新の規模 | アクション | 目安 |
|---------|-----------|-----------|------|
| 通る | パッチ（例: 15.2.3 → 15.2.4） | リリースノートに問題なければマージ | 5〜10 分 |
| 通る | マイナー（例: 15.2 → 15.3） | マージ → ステージングで目視確認 | 10〜15 分 |
| 通る | メジャー（例: 15 → 16） | マージ → ステージングでしっかり確認、必要なら改修 | 1〜数時間 |
| 落ちる | すべて | 原因調査。互換性問題があれば改修が必要 | 30 分〜 |

> **パッチ・マイナー・メジャーの見分け方**: 多くのライブラリは **メジャー.マイナー.パッチ** 形式です。PR タイトルの「from → to」で、どの位置の番号が変わったかを見て判断します。
